# AGENTS.md

本文件适用于整个仓库。任何自动化代理或贡献者开始工作前，都必须先阅读本文件、`README.md` 和 `docs/PLAN.md`。

## 1. 项目目标

构建一款面向公开用户的竖屏像素风网页跑酷游戏。首版的核心体验是：无需账号或教程，打开链接后 3 秒内可开始；玩家只控制跳跃；一局能完整经历开始、挑战、失败和一键重开。

实施时按以下优先级判断取舍：

1. 操作响应、判定一致和不存在随机死局。
2. 手机端可用性、首屏速度和稳定帧率。
3. 角色、平台和危险物的视觉可辨识度。
4. 美术丰富度、额外内容和发布后功能。

`docs/PLAN.md` 是产品范围和体验目标的主要依据；本文件负责把它转化为工程约束。如果二者冲突，先停止扩大实现范围，并在同一变更中记录需要确认的决策。

## 2. 当前阶段与默认决策

项目当前已完成 M1 和 M2 工程校准，iPhone Safari 最新修复回归通过，Android 验证按产品决定保留到 M4；当前进入 M3 视觉垂直切片阶段。除非用户另有明确要求，按以下默认值推进：

- 首版是 Cloudflare Pages 上的纯静态单机游戏，不实现服务器、登录、昵称或全球排行榜。
- 使用 Vite、严格模式 TypeScript、Canvas 2D 和原生 DOM/CSS，不引入大型游戏引擎或 UI 框架。
- 首发界面使用简体中文；所有用户可见文案集中管理，避免散落在游戏逻辑中，为英文版预留结构。
- 主角暂用橘白小猫；名称、域名、最终配色和正式素材都应当可替换，不得成为核心逻辑依赖。
- 手机逻辑视口基准为 `390 × 844`；桌面端保持 844 逻辑高度并扩展横向视野，目标宽高比约 `1.25:1`。响应式视口不得拉伸画面或改变物理单位。
- 默认使用 `npm`。一旦仓库出现锁文件，以锁文件对应的包管理器和已提交脚本为准，不混用锁文件。
- 不为了未来的排行榜预建 Workers/D1。`wrangler.jsonc` 只在 M5 真正启动时加入。
- “离线缓存”在 MVP 中指合理的静态资源缓存检查；除非需求明确，不擅自把项目扩展为完整 PWA。

## 3. MVP 边界

必须包含：

- `ready → playing → paused → gameOver` 状态流，以及显式继续和一键重开。
- 自动奔跑、单次跳跃、重力、落地、边缘跌落、失败判定。
- 可复现且保证可达的平台生成、向下推进的相机和分段难度曲线。
- 当前分数、本地最高分、版本化设置存储。
- 键盘、鼠标、单指触控输入；失焦/隐藏自动暂停。
- 静音、减少动态效果、安全区和横屏提示。
- 原创或许可明确的像素素材、基础分享信息和 Open Graph 图。

未经明确授权，不实现二段跳、长短按变高度、账号、排行榜、广告、付费、多角色、剧情或关卡编辑器。实验性功能必须默认关闭，不得复杂化核心路径。

## 4. 目标目录与职责

搭建项目时优先形成以下边界；只有在代码量确实需要时再继续细分：

```text
public/                 favicon、OG 图、无需打包处理的音频
src/
  game/                 状态机、固定步长模拟、物理、碰撞、相机、计分、平台生成
  render/               Canvas 渲染、精灵、调色板和视差层
  input/                DOM 事件到语义动作的映射
  ui/                   开始、暂停、失败、设置、提示等 DOM 界面
  storage/              版本化 localStorage 读写与损坏数据恢复
  content/              集中的用户文案；需要时再引入完整 i18n
  main.ts               组装依赖和启动应用，不承载游戏规则
tests/                  与 src 职责对应的确定性单元/集成测试
docs/                   计划、设计决策、素材授权和发布清单
```

规则层不得直接查询 DOM、读写 `localStorage` 或播放声音。渲染层只读取游戏快照，不修改世界状态。输入层只发出 `start`、`jump`、`pause`、`resume`、`restart` 等语义动作，不直接改变角色坐标。

## 5. 核心工程约束

### 游戏循环与状态

- 全应用只能有一个 `requestAnimationFrame` 所有者；暂停、重开和热更新后不得叠加循环或事件监听器。
- 渲染使用 `requestAnimationFrame`，模拟使用固定时间步长和累加器；页面恢复时限制可消费的最大 delta，禁止一次补算长时间后台帧。
- 每个输入都必须经过状态机过滤。暂停、游戏结束和 UI 操作期间不得触发跳跃。
- `visibilitychange`、窗口失焦或关键键盘焦点丢失时自动进入暂停；返回后必须由用户显式恢复。
- 重开必须从干净的会话状态开始，同时保留合法的最高分和设置。

### 坐标、物理与碰撞

- 游戏规则只使用逻辑坐标和世界坐标；CSS 尺寸、设备像素比及安全区只影响显示和输入映射。
- 桌面视口可以展示更多世界，但不能改变平台位置、碰撞、计分或随机序列；横向生成缓冲区必须随逻辑视口宽度扩展。
- 物理常量集中定义，单位和含义明确。不要在渲染、生成器或 UI 中复制速度、重力和尺寸魔法数。
- 落地碰撞需要结合上一帧位置，只在角色向下运动并从平台上表面穿越时成立，避免侧面吸附和从下方穿透后错误落地。
- 对快速运动使用扫掠/连续判定或等效方法，不能依赖“当前帧矩形重叠”碰运气。
- 相机移动不得改变世界中的物理位置；屏幕坐标应由世界坐标经过相机转换得到。
- 垂直相机只能平滑追踪小猫已安全落地的平台高度，不能脱离地形自行累积滚动；长局面中小猫和当前屋顶必须保持在可玩纵向区域。

### 平台生成与难度

- 所有运行时随机性来自可注入、可记录的种子；失败场景必须能用同一随机种子复现。
- 生成下一平台前，根据当前跳跃参数、水平速度、滞空时间、平台宽度和安全余量计算可达区间，再在区间内采样。
- 生成器必须提供确定性测试：大量种子下每一步都有至少一个合法落点；发现失败时输出种子和关键参数。
- 0–150、150–500、500+ 分三个阶段的数据集中配置。提高难度不能突破可达性约束，也不能只靠不可辨识的视觉干扰。
- 分数来自前进距离与成功落地奖励；同一模拟输入与种子必须得到相同结果。

### 输入、UI 与可访问性

- 支持 `Space`、`ArrowUp`、`W`、画布左键和第一根手指轻触。屏蔽按键自动重复造成的重复动作。
- 只有游戏区域处于合适状态时才拦截默认行为；表单、按钮和设置交互不得触发跳跃。
- 使用 Pointer Events 时明确处理 pointer capture、取消和多指输入。游戏区配置必要的 `touch-action`，不要全局破坏页面缩放和滚动。
- 交互目标至少约 44 CSS px，键盘焦点清晰，文字对比度可读。Canvas 内的重要信息必须在 DOM 中有等效状态或标签。
- `prefers-reduced-motion` 应关闭或减弱屏幕抖动、闪烁、粒子和大幅视差，不能改变碰撞与计分。
- 音频必须在用户手势后启用；静音偏好持久化。音频失败不能阻止游戏开始。

### 存储、素材与性能

- `localStorage` 数据包含 schema 版本；解析、类型检查或写入失败时回退安全默认值，不让异常中断游戏。
- 最高分只接受本局正常结束产生的有限、非负值。不要把本地数据当成可信的服务端成绩。
- 只使用原创素材、项目自有素材或许可清楚且已记录来源的素材。禁止临摹或复制现有商业游戏角色、名称和资源。
- 像素图关闭平滑并按整数或清晰的等比策略显示；高 DPI backing store 与逻辑分辨率分离。
- 高频循环中避免每帧创建临时对象、查询 DOM 或重复解码资源。资源加载失败时提供不会破坏玩法的占位或降级。

## 6. 分阶段实施计划

每个里程碑必须先满足验收条件，再进入下一阶段；视觉工作不能掩盖规则缺陷。

### M1：可玩灰盒

1. 初始化 Vite + TypeScript，建立基础页面、逻辑画布缩放和开发脚本。
2. 实现纯逻辑状态机、单一循环所有者、固定步长模拟和输入动作层。
3. 实现角色运动、一个平台、连续落地判定、失败与干净重开。
4. 加入确定性平台生成、相机推进、自动暂停和基础调试信息。
5. 为状态转换、物理、碰撞、输入去重、生成器可达性和重复重开补测试。

验收：无正式素材也能连续玩 3 分钟；键盘、鼠标和触控都能完成开始、跳跃、暂停、恢复、失败和重开；不存在叠加循环。

### M2：手感与难度

1. 集中配置并试玩校准跑速、跳跃、重力、平台范围与安全余量。
2. 实现三个分数阶段、计分、速度曲线、种子记录和本地最高分。
3. 扩大种子测试与边界输入测试，保留失败样本为回归用例。
4. 用真实手机尺寸验证初始安全区和首局可理解性。

验收：所有自动生成平台满足可达约束；同种子与同输入可复现；新玩家无需阅读长教程即可理解。

### M3：美术、音效与可访问性

1. 先建立调色板、精灵尺寸、动画帧率和素材授权记录，再替换灰盒。
2. 添加前中远景但保持玩法轮廓清楚；实现静音、减少动态、清晰焦点和安全区。
3. 集中界面文案，补全加载、暂停、失败、横屏和资源失败状态。
4. 在 360px 宽手机、高 DPI 桌面和低性能设备上检查清晰度及内存增长。

验收：小屏可辨、所有素材权属明确、关闭声音或减少动态后核心体验完整。

### M4：测试与发布

1. 补页面元数据、favicon、OG 图、错误/断网提示和生产配置。
2. 运行构建、类型、单测、浏览器和移动实机测试；记录结果与已知限制。
3. 建立 Cloudflare Pages 预览部署，确认构建产物目录、缓存、HTTPS 和回滚步骤。
4. 预览环境通过后再发布生产域名；域名尚未确定时不得硬编码。

验收：目标浏览器均可完成整局；预览和生产路径可玩；分享信息、移动操作和重开正常。

### M5：发布后决策

只有收到明确需求和数据依据后才设计排行榜。开始前先补 API、隐私、内容治理、限流、Turnstile、异常分数和运维可观察性方案，不复用 `localStorage` 信任模型。

## 7. 测试与质量门槛

初始化项目时至少提供以下脚本，名称若因工具链调整需同步更新本文件和 README：

```bash
npm run dev
npm run build
npm run typecheck
npm test
```

若引入 lint 或端到端工具，再提供 `npm run lint`、`npm run test:e2e`。不要添加没有实际配置或长期失败的占位脚本。

每次变更按风险执行最小充分验证：

- 规则、物理、生成器、计分或存储变更：类型检查 + 相关单测 + 全量单测。
- 输入、状态或 UI 变更：上述检查 + 键盘/指针关键路径浏览器验证。
- 渲染、缩放或适配变更：上述检查 + 360×640、390×844、430×932 和至少一个桌面视口的截图/实测。
- 构建、资源或发布变更：生产构建 + 本地预览 + 直接打开关键路由/刷新检查。

关键回归用例至少覆盖：

- 暂停、恢复、失败和连续重开不会创建多个循环。
- 长按键、快速连点、多点触控和 UI 点击只产生合法动作。
- 切后台很久再返回不会物理跳帧，且不会自动继续。
- 高速下不穿透平台，侧面/底部接触不算落地。
- 同种子生成相同平台；批量种子不存在不可达的下一跳。
- 存储为空、旧版本、损坏、超额或不可写时安全降级。

不要用“测试通过”代替实际操作验证，也不要仅凭截图判断输入与碰撞手感。

## 8. 变更工作流

1. 开始前确认当前里程碑和用户请求范围，检查工作区已有改动，不覆盖无关工作。
2. 优先交付最小的端到端可玩切片，再逐步增强。避免一次提交同时重写物理、输入、渲染和生成器。
3. 对影响手感的常量，说明单位、默认值和调校理由；对架构或产品默认值的改变，在 `docs/` 中记录简短决策。
4. 新依赖必须说明必要性、体积和维护成本；标准 Web API 足够时不引入依赖。
5. 完成后报告修改内容、执行过的验证、尚未验证的设备/浏览器以及剩余风险。
6. 每项完成的变更在相关质量门槛通过后，必须创建范围明确的 Git commit，并推送到已配置的 GitHub remote；不得把已完成工作只留在本地工作区。
7. 开始工作时必须检查本地仓库、当前分支、工作区改动、`origin` 和 GitHub 认证。缺少 `.git`、远端、推送权限或必要的仓库可见性决策时，不得猜测或擅自创建错误远端；应把它作为交付阻塞明确报告并请求用户决定。
8. 工作区有混合改动时只暂存本项涉及的文件，不得静默提交或推送用户的无关改动。交付说明必须列出分支、commit 和 push 结果。

以下情况必须暂停并请求产品决策：超出 MVP 的服务端或用户数据收集；改变单次跳跃核心操作；使用权属不明的素材；需要付费服务或正式域名操作；待确认事项会造成大量返工。其余可逆的视觉或内部实现细节按本文件默认值推进。

## 9. 完成定义

一项工作只有同时满足以下条件才算完成：

- 行为符合 `docs/PLAN.md`、本文件及当前里程碑，不夹带未授权范围。
- 关键逻辑具有确定性测试，相关质量命令通过。
- 移动端和桌面端核心路径仍可用，错误与存储失败有降级。
- 没有遗留调试开关、未授权素材、控制台异常或重复循环/监听器。
- 文档、脚本、配置和实际行为一致；任何未验证项在交付说明中明确列出。
- 已按变更工作流创建并推送对应 Git commit；若 GitHub 基础条件缺失，则已明确标记阻塞且没有伪称完成发布。
