# 真实手机验收手册

本手册用于关闭 M2 的物理设备验收。响应式浏览器只能验证布局，不能替代真实触屏、移动浏览器地址栏、安全区、后台生命周期和设备性能。

## 1. 通过规则

至少使用以下两台设备：

- 一台近五年内的 iPhone，使用最新可用的 Safari。
- 一台常见安卓手机，使用最新可用的 Chrome。

两台设备都完成 A–F 测试，并且没有 P0/P1 缺陷，才能将 M2 标记为完整通过。只有一类设备通过时，结论必须写成“部分通过”。

## 2. 建立手机访问地址

1. 手机与开发机连接同一个 Wi-Fi；临时关闭会隔离局域网的 VPN、代理或访客网络。
2. 在项目根目录运行：

   ```bash
   npm install
   npm run dev -- --host 0.0.0.0
   ```

3. 终端会显示类似 `Network: http://192.168.x.x:5173/` 的地址。
4. 在手机浏览器输入完整的 Network 地址。不要使用 `localhost`，它在手机上指向手机自己。
5. 测试结束后在终端按 `Ctrl+C` 停止服务器。

无法访问时依次检查：

1. 手机和开发机是否在同一个非访客 Wi-Fi。
2. macOS 防火墙是否允许 Node.js 接收入站连接。
3. 暂时关闭手机蜂窝网络，排除浏览器绕过 Wi-Fi。
4. 在开发机运行 `ipconfig getifaddr en0` 核对 Wi-Fi 地址。
5. 不要把开发端口直接暴露到公网；跨网络测试使用 Cloudflare Pages 预览部署。

## 4. A–F 测试轮次

### A. 首次进入与理解成本

1. 关闭旧标签页，从新标签页第一次打开链接。
2. 不阅读项目文档，只看手机画面。
3. 记录从页面可见到第一次有效跳跃的时间。
4. 检查是否需要滚动、缩放或关闭遮挡物才能开始。

通过标准：

- 3 秒内看见主角、目标提示和开始入口。
- 一次轻触即可开始并完成第一次跳跃。
- 没有横向滚动、意外选中文字或双击缩放。
- 开始按钮、分数和主角未被刘海、灵动岛或底部手势条遮挡。

### B. 核心玩法三局

连续完成三局，记录分数、持续时间、失败原因和主观手感：

1. 第一局按直觉玩。
2. 第二局尝试较早起跳。
3. 第三局尝试贴近边缘起跳。

重点观察：

- 起跳是否立即响应，有没有“按了但没跳”或一次触摸跳两次。
- 落地是否符合画面，是否出现侧面吸附或穿过平台。
- 0–150 分是否足以建立节奏；150–500 分变化是否能感知但不突兀。
- 失败原因是否明确，重开是否只需一次点击。

通过标准：三局都能完成“开始 → 游戏 → 失败 → 重开”闭环，没有无法解释的失败和随机死局。

### C. 触控压力测试

每项执行 10 次：

- 在画布中央快速单击。
- 以约每秒 4–6 次速度快速连点。
- 两根手指近乎同时触碰画布。
- 一根手指按住时，第二根手指触碰。
- 在屏幕左右边缘附近轻触。
- 点击暂停、继续和重开按钮。

通过标准：只处理第一根有效指针；多指和连点不产生异常跳跃；UI 按钮不会同时触发跳跃；页面不被拖动或缩放。

### D. 页面生命周期

1. 游戏中切到另一个应用，等待 10 秒再返回。
2. 游戏中锁屏，等待 10 秒后解锁。
3. 游戏中打开通知中心或多任务界面再返回。
4. 暂停状态下重复上述步骤。

通过标准：离开页面时自动暂停；返回后停留在暂停界面；必须明确点击继续；不会补算后台时间、突然坠落或叠加循环。

### E. 视口与系统设置

1. 轻扫使浏览器地址栏收起、展开。
2. 竖屏转横屏，再转回竖屏。
3. 开启系统“减少动态效果”后重新加载。
4. 检查所有按钮的触控面积和安全区。

通过标准：地址栏变化不裁切核心 UI；横屏出现简洁提示；回到竖屏后状态正常；减少动态效果不改变物理和计分。

### F. 连续运行与性能

1. 从冷启动开始连续操作至少 3 分钟；允许失败后立即重开，但页面不能刷新。
2. 再连续完成三局，观察后两局是否比第一局更卡。
3. 记录明显掉帧、输入延迟、发热、音画不同步和浏览器自动刷新。

通过标准：无崩溃、白屏或自动刷新；后续局没有持续恶化；触摸延迟稳定；设备可以温热，但不应在几分钟内明显烫手或强制降频。

## 5. 缺陷等级

| 等级 | 含义 | 示例 |
| --- | --- | --- |
| P0 | 无法继续测试或有设备风险 | 页面打不开、持续崩溃、浏览器死锁 |
| P1 | 核心闭环损坏 | 无法跳跃、无法重开、后台返回直接失败、随机死局 |
| P2 | 明显影响体验但可绕过 | 偶发误触、按钮被部分遮挡、难度突变 |
| P3 | 轻微视觉或文案问题 | 像素模糊、提示换行不理想 |

P0/P1 必须修复并在原设备复测。P2 必须决定“修复后复测”或“记录为已知限制”。P3 可进入 M3/M4 集中处理。
